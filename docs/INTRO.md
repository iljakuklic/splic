# Introduction to Splic

Welcome to **Splic**, an experimental programming language for writing zero-knowledge virtual machine (zkVM) code. Splic bridges the gap between high-level abstractions and low-level control, giving you the power of a strongly-typed, dependently-typed language alongside precise governance over the generated bytecode.

## The Problem

Writing zkVM bytecode directly is tedious and error-prone. Existing approaches face a dilemma:

- **Low-level languages** (assembly-like bytecode) give you complete control but demand manual bookkeeping of every operation, making code hard to maintain and verify.
- **High-level languages** (Python, Rust, etc.) abstract away the details beautifully, but you lose fine-grained control over the generated proof—often a critical requirement in cryptographic systems where every instruction matters.

Splic sits at the sweet spot: **high-level abstractions for productivity, with direct control over generated zkvm bytecode**.

## Two-Level Types: Principled Metaprogramming

Splic uses **two-level type theory (2LTT)**: a principled approach that cleanly separates compile-time (meta-level) computation from runtime (object-level) code generation. This is a **principled alternative to macros**—providing type-safe code generation without the fragility and complexity of macro systems.

- **Meta-level**: A purely functional, dependently-typed language where you write programs and express invariants as types. The typechecker verifies these invariants at compile time.
- **Object-level**: A low-level language exposing zkVM primitives. You see exactly what bytecode will be generated.
- **Quotations and Splices**: Bridge the two levels safely. `#(expr)` quotes meta-level code into object code; `$(expr)` splices object code back into the meta-level. This enables **type-safe metaprogramming**—manipulation and composition of bytecode with full static guarantees.

This approach lets you write efficient, optimized code generators that produce proven-correct object code.

## Dependent Types: Expressing Invariants

Dependent types let you encode rich specifications directly in types. Instead of asserting properties at runtime, you prove them at compile time.

Examples:
- Encode the length of a vector in its type: `Vec n` means a vector with exactly `n` elements.
- Express algorithmic invariants: "after this step, the balance is non-negative."
- Build certified data structures with invariants baked into the type.

This approach enables **high-assurance software**: if the code typechecks, the invariants hold by construction.

## Fine-Grained Code Control

Unlike many high-level languages, Splic does not hide details. The object-level language is explicit about control flow and operations. This matters for zkVMs where:

- **Proof size matters**: Unnecessary instructions inflate the proof.
- **Optimization visibility**: You can see (and influence) what code gets generated.
- **Resource costs are explicit**: Operations have measurable costs; the language makes them visible.

## Language Features

### Core Features (Implemented)

- **Dependent types** with full type checking
- **Quotations** (`#(...)`) to produce object-level code from meta-level expressions
- **Splices** (`$(....)`) to embed object-level code into meta-level programs
- **Pattern matching** on values with exhaustiveness checking
- **Function definitions** separated into meta-level (`fn`) and object-level (`code fn`)
- **Primitive types**: `u0`, `u1`, `u8`, `u16`, `u32`, `u64`
- **Primitive operations**: arithmetic (`+`, `-`, `*`, `/`), comparison (`==`, `!=`, `<`, etc.), bitwise (`!`, `&`, `|`)
- **Let bindings** with optional type annotations
- **Type inference** in function bodies

### Planned Features

- **User-defined types** and algebraic data types (ADTs)
- **Object-level control flow** (loops, conditionals with proof semantics)
- **Closure-free object language** for more predictable compilation
- **Phase polymorphism** to reduce annotation burden
- **Effect handling** for I/O and system interaction
- **Implicit arguments** (once patterns stabilize)

## A Concrete Example

Here's a classic example: computing powers with a compile-time exponent. The meta-level function `power` recursively builds object-level code; when called with a concrete exponent during compilation, it fully unrolls into straight-line code.

```splic
fn power(x: [[u64]], exp: u64) -> [[u64]] {
    match exp {
        0 => #(1),
        1 => x,
        exp => {
            let exp2: [[u64]] = #{
                let x2 = $(power(x, exp / 2));
                x2 * x2
            };
            let odd: u64 = exp & 1;
            match odd == 1 {
                0 => exp2,
                1 => #($(exp2) * $(x)),
            }
        }
    }
}

code fn pow5(x: u64) -> u64 {
    $(power(#(x), 5))
}
```

The type `[[u64]]` means "code that computes a `u64`". The `power` function builds code recursively:
- When `exp == 5`, the meta-level evaluates `power(#(x), 5)` at compile time
- The recursive calls unroll, computing `exp/2`, then squaring, doubling as needed
- The result: a `code fn pow5` that directly computes `x^5` with no recursion overhead

The generated object-level code looks like:

```splic
code fn pow5(x: u64) -> u64 {
    let x2 = x * x;
    let x4 = x2 * x2;
    let x5 = x4 * x;
    x5
}
```

## Unlimited Weirdness Budget

Splic is **experimental**. We are not constrained by backward compatibility or conservative design choices. The goal is to explore what becomes possible when programmers have both **high-level abstraction power** and **low-level control over code generation**.

This means:

- **Syntax and semantics may change** as we learn what works well.
- **Novel features** may be introduced to solve zkVM-specific problems in elegant ways.
- **Unconventional ideas** are welcome if they improve the development experience or code quality.

If you are drawn to cutting-edge language design and don't mind the research-level instability, Splic is for you.
