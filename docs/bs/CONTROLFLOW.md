# Control Flow Constructs

This document outlines proposed approaches to control flow in Splic, discussing trade-offs and open questions.

## Approaches

### 1. Standard Approach

The meta-level uses standard functional constructs for control flow: pattern matching for branching, recursion for looping.

The object-level uses conventional structured control flow: if-statements, while loops, etc.

### 2. Functional Goto

A control flow construct inspired by SSA IR representation. A function is split into named basic blocks, and control transfer happens through `goto` to a basic block. Goto and other control flow constructs (e.g. `return`) are only permitted in tail position.

Example of a simple for loop:

```
entry(n_iters):
    match n_iters {
        0 => goto end(),
        _ => goto body(0),
    }

body(i):
    // loop body here
    match i == n_iters {
        true => goto end(),
        false => goto body(i + 1),
    }

end():
    // something
```

With pattern matching directly on arguments:

```
entry(0):
    goto end()

entry(n_iters):
    goto body(0)

body(i):
    // loop body here
    match i == n_iters {
        true => goto end(),
        false => goto body(i + 1),
    }

end():
    // something
```

Basic blocks behave as pure functions—iteration is achieved through tail-calling the block with different arguments.

### Termination Checking

With functional goto, each basic block is effectively a separate function. Standard structural recursion checks apply to individual blocks. The key question is whether the language permits partial functions at all, and if so, how:

- **Total-only meta-level**: All recursive goto-loops must pass termination checking, same as any recursive function
- **Allow partial functions**: Like Coq's `Program Fixpoint` or Idris, termination becomes a programmer responsibility for certain definitions

This is not unique to functional goto—the same trade-off exists with standard recursion. The difference is more visible syntactically.

## Trade-offs

### Functional Goto: Advantages

- **Uniformity**: Can be used at both meta and object levels, allowing more functions to be phase-polymorphic
- **Low-level control**: Direct mapping to SSA, giving programmers precise control over generated bytecode
- **Desugaring target**: Standard loop constructs can desugar to functional goto
- **CPS correspondence**: One-to-one correspondence with continuation-passing style (with restrictions), which is well-understood
- **Purely functional with IR mapping**: Combines the benefits of functional programming with direct mapping to many IR representations

### Functional Goto: Disadvantages

- **Variable scoping complexity**: Either all variables must pass through block arguments (verbose), or local variables follow CFG dominator tree visibility rules
- **Dependent types interaction**: Return types must unify across all goto targets from a given block; with dependent types where the result type depends on the branch, this becomes nontrivial
- **Staged compilation interaction**: How do gotos behave when meta-level functions are quoted into object-level code?
- **Ergonomics**: More verbose than `for i in 0..n { ... }`
- **Tooling complexity**: IDE features like go-to-definition, rename refactoring, and code analysis require understanding of dominator-tree scoping — more complex than standard lexical scoping
- **Backend mismatch**: Some zkVMs use WebAssembly (or are modelled after it) which enforces structured control flow; converting from source-level SSA-like format would be required, losing the direct mapping advantage

### Standard Approach: Advantages

- **Simplicity**: Well-understood, matches existing functional languages
- **Dependent types**: Works cleanly with dependent type theory—no additional complexity
- **Ergonomics**: Familiar, concise syntax for loops and branching
- **Separation of concerns**: Meta-level stays for high-level reasoning, object-level for low-level control
- **Proven**: Similar approach used in many dependently typed languages

### Standard Approach: Disadvantages

- **Less control**: Programmer has less direct influence over generated bytecode
- **Phase-specific**: Meta-level and object-level require different control flow constructs
- **Desugaring**: Standard constructs must desugar to something—need to decide what the target representation is

## Additional Considerations

### Exceptions

- Meta-level: No exceptions; errors handled via `Result<_, _>` types
- Object-level: Assertions fail the entire program, which is typical for zkVMs—goto does not significantly complicate this

### Open Questions

- Should functional goto be object-level only, or both levels?
- If both levels: how to handle the dependent type interaction?
- If object-level only: is the meta-level too restrictive for phase-polymorphic code?

## Summary

| Aspect | Standard | Functional Goto |
|--------|----------|-----------------|
| Dependent types | ✓ Clean | Complex |
| Implementation complexity | Low | High |
| Low-level control | Indirect | Direct |
| Phase-polymorphism | Limited | Full |
| Ergonomics | Familiar | Verbose & Weird |

A pragmatic path forward may be to start with the standard approach—functional constructs at meta-level, structured control flow at object-level—and add functional goto later if the need arises. This defers complexity while preserving the option.
